## DnD 跑团感的具体含义

做这件事之前必须对齐"DnD 跑团感"是什么。对我们产品来说, 它包括五个可验证要素:

1. **有 GM (Game Master)** — 世界有一个会讲故事的"看不见的叙述者"。对我们是 AI Storyteller + Tier 3 模型扮演 GM 声音
2. **骰子与不确定性** — 关键动作有明确的掷骰和成功率, 玩家能看见"DC 15, 你掷了 12, 失败"这样的机制暴露
3. **角色深度** — 每个 NPC 有完整 backstory, 玩家问起来能讲出自己的人生
4. **选择有后果** — 玩家一个决定会让某 NPC 24h 后状态变化, 且变化会被其他 NPC 记住
5. **emergent narrative** — 系统产出的故事超出剧本, 带来"我们玩的是同一个世界但经历不同"的感觉

**反面案例** (我们要避免的):
- 华丽 UI 但 NPC 都是复读机 → 没 DnD 感
- 大量预设任务线 → 是 JRPG, 不是跑团
- 骰子全隐藏 → 玩家感受不到机制

## Stage A — 无玩家虚拟世界模拟器

### 目标
世界自转. 每天自发产出 legend log, 团队读起来像读章回体小说.

### A.0 Foundation (2-3 周)

- [ ] **A.0.1** 代码仓库改造: 从 `site-zero` 剥离, 建立 `world_packs/` **独立插件架构** (`scp/`, `cthulhu/`, `liaozhai/`), 每个 pack 自包含 persona + 事件表 + storyteller 人格 + 世界观 tile 类型。运行时可以: (a) 单 pack 独立启动 (一个纯聊斋世界), (b) 多 pack 共享 world space 启动 (三世界观并存), (c) pack 之间通过可选的 `cross_pack_events.yaml` 定义交织事件
- [ ] **A.0.2** Persona schema 统一 (见 [stat-machine-design.md](stat-machine-design.md) Agent 数据结构)
- [ ] **A.0.3** World state storage: PostgreSQL + pgvector + Redis
- [ ] **A.0.4** 国内 LLM 供应商对接: DeepSeek V3 API (Tier 3), Qwen2.5-7B 自托管 (Tier 2)
- [ ] **A.0.5** 基础可观测层: 每 tier 调用量/延迟/成本日报

### A.1 Tier 1 Stat Machine (4-6 周) ← **核心工作**

- [ ] **A.1.1** AI Storyteller 实现 (tension curve + cooldown + tile 事件池)
- [ ] **A.1.2** Agent Stat Machine (属性 + 立场 + 关系 + 道具 + legend_log)
- [ ] **A.1.3** Historical Figure Registry (晋升/降级逻辑)
- [ ] **A.1.4** 事件表撰写 (300-500 条, 每世界观 100+)
  - SCP 事件池: 收容失效 / 报告编写 / 研究突破 / D-class 测试 ...
  - 克苏鲁事件池: 低语降临 / 教团仪式 / 调查员发疯 / 古籍发现 ...
  - 聊斋事件池: 市井偶遇 / 赶考借宿 / 妖精修行 / 人鬼情缘 ...
- [ ] **A.1.5** Legend log template 系统 (每类事件有模板化叙事输出)
- [ ] **A.1.6** 跨 worldview 事件 (SCP 特工追查一个聊斋狐妖, 克苏鲁低语污染 SCP 收容站, etc.)

### A.2 Seed Persona 库 (与 A.1 并行, 4-6 周)

- [ ] **A.2.1** 三世界观各采 50 个角色 (total 150)
- [ ] **A.2.2** 每个角色编写 persona card (< 500 字) + 属性表 + 初始关系
- [ ] **A.2.3** **合成 interview transcript** (Tier 3 模型双向扮演, 每角色 ~1 小时对话量)
- [ ] **A.2.4** Transcript 导入 vector DB 作为初始长期记忆
- [ ] **A.2.5** Historical figures 名单敲定 (每世界观 10-20 个"世界级角色")

### A.3 Tier 2 Legend 润色 (2-3 周)

- [ ] **A.3.1** Qwen2.5-7B 自托管部署 (vLLM)
- [ ] **A.3.2** 重要度评分函数 (见 stat-machine-design.md)
- [ ] **A.3.3** Tier 2 润色 prompt 模板 + 输出解析
- [ ] **A.3.4** Fallback 机制 (Tier 2 失败降 Tier 1)

### A.4 Tier 3 Spotlight + Debate Phase (3-4 周)

- [ ] **A.4.1** DeepSeek V3 对接 + prompt cache 配置
- [ ] **A.4.2** Spotlight event 识别器 (score ≥ 0.6 触发)
- [ ] **A.4.3** Debate Phase 实现 (见 [flow-loops.md](flow-loops.md) Flow 4)
- [ ] **A.4.4** 每日 reflection 层 (压缩 raw events 成 narrative)

### A.5 内部 Dashboard (与 A.3/A.4 并行, 2-3 周)

- [ ] **A.5.1** Web dashboard (Next.js / Streamlit)
- [ ] **A.5.2** 视图: tile 地图 / agent 列表 / 关系图 / legend 时间线
- [ ] **A.5.3** "章回体小说模式": 每日 digest 把 legend 渲染成可读文本
- [ ] **A.5.4** 成本/性能监控面板

### Stage A Exit Criteria

必须全部达标才能进 Stage B:

- ✅ 在一台 4×A10 GPU 机器 + DeepSeek API 上跑 **100 虚拟日** 不中断
- ✅ 日均产生 **500+ legend events**, 其中 20+ 为 spotlight 级
- ✅ 团队内 **5 个非开发人员** 读一周 legend digest 能记住 10 个以上 NPC 的故事
- ✅ 盲测: 读者分不清 legend digest 是人写的还是 AI 生成的 (Turing-level 叙事)
- ✅ **Tier 1 占 call 数 ≥ 95%**, Tier 3 日均花费 < ¥200

> **如果 A.X 结束时团队自己都觉得"世界很无聊"——不要进 Stage B。Pivot 或调整事件池直到达标。这是所有 living world 产品的生死线。**

## Stage B — 有玩家的虚拟世界模拟器 (DnD 感)

### 目标
10 人内测玩家像玩文字跑团一样玩. Web + 地图, 无 AR 无 LBS.

### B.1 玩家身份与客户端 (3-4 周)

- [ ] **B.1.1** 玩家账户体系 (手机号 + 实名制预留)
- [ ] **B.1.2** Web 客户端 (Next.js): chat UI + 简易地图 + 骰子动画
- [ ] **B.1.3** 玩家角色创建 (职业/属性/起始道具, 完全 DnD 风格)
- [ ] **B.1.4** "世界选择": 玩家选 SCP 特工/克苏鲁调查员/聊斋书生 身份入场

### B.2 玩家-Agent 交互 (4-6 周) ← **DnD 感的核心**

- [ ] **B.2.1** **文字对话系统**: 玩家自由输入, agent 基于自己 persona + memory 回复
- [ ] **B.2.2** **骰子机制外显**: 关键动作 (说服/潜行/施法/战斗) 展示 DC 和掷骰结果
- [ ] **B.2.3** **技能检定**: 玩家属性对掷骰加成, 公式对玩家透明
- [ ] **B.2.4** **选择有后果**: 玩家每个决定写入 world state, 影响 agent 未来状态
- [ ] **B.2.5** **Per-player memory scope**: agent 对不同玩家有不同记忆
- [ ] **B.2.6** **GM 叙述声**: Tier 3 扮演 "世界的描述者", 每个玩家动作后给环境反馈

### B.3 世界感知反馈 (3-4 周)

- [ ] **B.3.1** **离线事件通知**: 玩家离线时 agent 给玩家"留言" (信件/梦境/线索)
- [ ] **B.3.2** **回归追述**: 玩家回来时 Tier 3 生成"你走后这几天发生了什么"
- [ ] **B.3.3** **关系变化可视化**: UI 上展示玩家与各 agent 的好感度变化
- [ ] **B.3.4** **因果链可追溯**: 玩家能回放"我上次某决定导致了现在的什么"

### B.4 多 world pack 支持 (2-3 周)

- [ ] **B.4.1** 玩家进入前选择"世界配置": 单 pack 纯净体验 (如纯聊斋) / 多 pack 混合体验
- [ ] **B.4.2** 跨世界观任务 (仅混合模式): SCP 特工调查一个聊斋狐妖
- [ ] **B.4.3** 身份切换: 玩家可以在不同世界观创建不同角色, 互不干扰
- [ ] **B.4.4** 调性一致性: 三世界观的文字渲染风格明显有别 (SCP 报告体 / 克苏鲁哥特散文 / 聊斋文言白话)

### B.5 跑团工具 (2-3 周)

- [ ] **B.5.1** **角色卡**: 玩家随时查看自己 + 已接触 agent 的完整档案
- [ ] **B.5.2** **世界日志**: 玩家视角的事件编年史
- [ ] **B.5.3** **手札系统**: 玩家可写个人笔记/推论, 系统会把推论反向注入 agent 对话判断
- [ ] **B.5.4** **地图交互**: 玩家可选 tile 移动 (虚拟世界, 非真实 LBS)

### Stage B Exit Criteria

- ✅ **10 个内测玩家, 30 天留存 ≥ 40%** (DnD 类单机 benchmark)
- ✅ 单次 session 平均时长 ≥ **25 分钟**
- ✅ **80% 玩家能说出"我和某 NPC 的故事"**
- ✅ 玩家盲测: "这个感觉像 DnD 跑团/太吾/鬼谷的精神续作" 得分 > 4/5
- ✅ 经济模型: 单玩家月成本 (Tier 3 调用) < ¥30

## Stage C — AR 世界模拟器

### 目标
把 Stage B 的体验搬到移动端 + 真实地理位置, **但不加收集对战**. 保持 DnD 感.

### C.1 LBS 技术栈落地 (3-4 周)

- [ ] **C.1.1** 地图 SDK 对接: **高德地图** (国内覆盖最好) 或腾讯地图
- [ ] **C.1.2** **坐标系处理**: WGS-84 ↔ GCJ-02 转换 (法规要求)
- [ ] **C.1.3** 定位融合: GPS + Wi-Fi 指纹 + IMU, 目标精度 10-30 m
- [ ] **C.1.4** Tile-to-geography 映射: H3 resolution 9 (~100m grid)
- [ ] **C.1.5** 反作弊: GPS 伪造检测 + 速度校验 + 设备指纹

### C.2 移动原生 app (6-8 周)

- [ ] **C.2.1** **iOS 先行** (Swift + ARKit + 高德 iOS SDK)
- [ ] **C.2.2** App 核心视图: 地图 + 附近 agent 列表 + 对话
- [ ] **C.2.3** 基础 AR 渲染: agent 以 2D sprite 贴在真实场景 (MVP 先不做 3D)
- [ ] **C.2.4** 性能优化: 电量 / 流量 / 发热
- [ ] **C.2.5** Android 版本 (ARCore 国内 + 小米/华为/OPPO SDK 兼容)

### C.3 物理位置触发逻辑 (2-3 周)

- [ ] **C.3.1** 玩家 physically 进入 tile → awake tier 激活
- [ ] **C.3.2** 退出 tile → 降级 dormant
- [ ] **C.3.3** **室内模式**: 无法外出时 (天气/夜晚/工作日) 允许"虚拟漫游"但奖励减半
- [ ] **C.3.4** 首城覆盖: 选一座城 (建议杭州/成都, 用户密度 + 政策相对友好), 人工规划 500-1000 个 tile 与 agent 分布

### C.4 移动 DnD 体验 (3-4 周)

- [ ] **C.4.1** 手机屏幕上**保持**骰子/属性检定/文字叙述的 DnD 感
- [ ] **C.4.2** 语音交互 (可选, 讯飞/火山 STT-TTS)
- [ ] **C.4.3** 触摸手势对应动作 (长按潜行 / 双击说服 等)

### Stage C Exit Criteria

- ✅ iOS + Android 双平台可用, 首城覆盖完成
- ✅ 玩家 physically 出门完成任务率 > 50% (否则说明游戏没真正"LBS 化")
- ✅ Stage B 的 DnD 感在手机上**没有丢**
- ✅ 30 天留存 ≥ 25% (移动游戏 benchmark)

## Stage D — AR 宝可梦玩法层

### 目标
在 Stage C 之上加"收集养成对战社交", 撑起日活和商业化.

### D.1 收集系统 (3-4 周)

- [ ] **D.1.1** **羁绊**机制: 玩家与 agent 建立深度关系后, 可"缔结羁绊" (不是"收服", 保留 DnD 感)
- [ ] **D.1.2** 羁绊带来能力: 该 agent 在玩家下次对话时优先响应, 提供属性加成
- [ ] **D.1.3** 羁绊等级: 1-5 星, 影响 agent 对玩家的信任深度
- [ ] **D.1.4** **不是抓捕**: 玩家不能"拥有" agent, agent 仍有自己人生, 只是和玩家有更强的纽带

### D.2 养成系统 (3-4 周)

- [ ] **D.2.1** 玩家可以"推动" agent 成长 (教授技艺 / 赠送道具 / 委托任务)
- [ ] **D.2.2** agent 的属性变化反哺玩家 (师傅变强, 学徒跟着牛)
- [ ] **D.2.3** 道具词条系统 (借鉴鬼谷八荒)

### D.3 对战系统 (4-6 周)

- [ ] **D.3.1** **PVE 剧情战**: 玩家带着羁绊 agent 挑战 spotlight 事件 (如 SCP 收容失效)
- [ ] **D.3.2** **PVP 博弈**: 两玩家的 agent 在 Debate Phase 中对立, 玩家一起决策
- [ ] **D.3.3** **不是数值互殴**: 战斗是"多方博弈推演", 靠属性 + 策略 + 羁绊等级, 这才是我们区别于宝可梦的地方
- [ ] **D.3.4** 战斗回合系统与文字跑团兼容

### D.4 社交与 UGC (4-6 周)

- [ ] **D.4.1** 玩家之间共享 tile 内的 agent, 可以看到彼此痕迹
- [ ] **D.4.2** 多玩家卷入同一 debate phase (更精彩)
- [ ] **D.4.3** UGC 通路: 玩家提交自创 agent/事件, 审核后进入公共池
- [ ] **D.4.4** 审核管线 (人工 + LLM 双层)

### D.5 商业化 (2-3 周)

- [ ] **D.5.1** 充值体系 (符合国内规范)
- [ ] **D.5.2** 付费点: 稀有 agent 相遇券 / 高 Tier3 调用权 / 外观/称号 / 月卡
- [ ] **D.5.3** **不做**: 强抽卡数值碾压, 与 DnD 感冲突

### Stage D Exit Criteria

- ✅ DAU/MAU ≥ 25% (活跃度稳定)
- ✅ 付费率 ≥ 3% (中国手游 benchmark)
- ✅ 用户 NPS ≥ 40
- ✅ **玩家自发产出优质 UGC ≥ 10% 内容**

## 关键风险与应对

| 阶段 | 最大风险 | 应对 |
|---|---|---|
| Stage A | 事件池写不出"戏剧感" | 请游戏设计/编剧背景成员主导事件表; 参考太吾模组社区 |
| Stage A | Tier 1 世界感觉"死板" | 加大 storyteller 扰动, 提高 historical figures 比例 |
| Stage B | 玩家觉得"就是个聊天机器人" | **强化骰子/检定机制外显**, 让机制可感知 |
| Stage B | 留存不到 40% | 很可能是叙事断层, 增强 reconnection hooks (离线信件) |
| Stage C | 首城 LBS 运营成本高 | 先小范围 (杭州某个区 2 km²) 验证再扩 |
| Stage D | 版号批不下来 | 提前 Stage B 阶段就启动申请; 备份方案: 先港澳台或东南亚版 |
| 全程 | 国内 LLM 厂商限流/涨价 | **双供应商策略**: DeepSeek + Qwen-Max, 随时切换 |

## 最关键的一条建议

**Stage A 的出口标准 (Turing-level 叙事 + 团队读 digest 上瘾) 如果一年内达不到, 就不要硬冲 Stage B-D**. 

living world 游戏的成功基因是: **模拟器本身已经好看**. Dwarf Fortress 是这样, 太吾/鬼谷是这样, Stanford Smallville 也是这样. 如果 Stage A 就失败, 加 AR、加宝可梦只是把无聊包装得更花哨, 玩家会在 3 天内抛弃产品.

反过来, 如果 Stage A 成功了, 后面 B/C/D 都是"锦上添花"——即便其中某一 stage 体验不到位, **底层的世界永远有看头**, 用户会持续回来.
